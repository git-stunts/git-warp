### Observer and Worldline Nouns

| Codebase Noun | Paper Concept | Alignment | Notes |
|---|---|---|---|
| `Worldline` (class) | Worldline — deterministic sequence `U₀ → U₁ → … → Uₙ` | **RED** | **The codebase `Worldline` is not a worldline.** In the papers, a worldline is the full sequence of states generated by ticks under a scheduling policy (WARP II, OG-I §2). In the code, `Worldline` is a thin read-handle that wraps a graph reference and a source descriptor. It calls `materialize()` and returns a snapshot. It has no history, no sequence, no BTR chain. It's a **projection handle**, not a worldline. |
| `WorldlineSource` (type) | Worldline selector — which worldline an observer projects | **RED** | **Not a source of worldlines.** This type selects WHICH worldline to observe: the canonical worldline (all writers merged), a hypothetical worldline (specific writer tips), or a single writer's isolated worldline. The observer then picks a tick (ceiling) on that worldline. This is not a frontier — frontiers are a writer concern. |
| `Observer` (class) | Observer — the projection half (π) of an optic | **RED** | **Narrower than the paper concept.** In OG-I, an observer is a structural five-tuple `S = (O, B, M, K, E)`. In the code, `Observer` implements only the projection `O` (glob match + expose/redact). The optics note confirms: "Observer is closer to the projection half of an optic." The full optic Ω = (π, φ, ρ, ω, σ) maps to the WRITER, not the observer — writers project, focus, rewrite, witness, and reintegrate. |
| `Aperture` (interface) | Aperture — the set of distinctions that survive projection (OG-I §4) | YELLOW | Correct concept, but the codebase `Aperture` is just `{ match, expose, redact }` — a glob filter. In OG-I, aperture is a formal measure over a distribution of distinctions (Def. 4 in OG-I). The codebase captures the intent but not the full formalism. Acceptable — the interface can grow. |
| `ObserverConfig` | Legacy alias for Aperture | GREEN | Just an alias. Correctly deprecated. |

### Writer-as-Optic Mapping

The WARP Optics note defines `Ω = (π, φ, ρ, ω, σ)`. This maps
cleanly to the WRITER, not the observer:

| Optic component | Writer operation | Codebase |
|---|---|---|
| **π** (projection) | Writer sees the graph through its frontier | `materialize()` at the writer's VV |
| **φ** (footprint/focus) | Entities the patch reads/writes/deletes/anchors | Footprint in `PatchBuilderV2` ops |
| **ρ** (rewrite) | The patch operations | `PatchBuilderV2.addNode()`, etc. |
| **ω** (witness) | Data sufficient for local reversibility | Contained within `TickReceipt` |
| **σ** (reintegration) | Commit the patch, advance the ref | `PatchController.commit()` |

**The Observer is just π** — projection. It doesn't rewrite, doesn't
own a frontier, doesn't produce witnesses. It picks a worldline and
a tick, then projects.

**The Writer is the full optic.** It owns a frontier (its causal
boundary), submits intents (rewrites), and advances the worldline.
Frontiers are a writer concern, not an observer concern.

This distinction matters for naming: `WorldlineSource` should not
reference frontiers. It selects which worldline to observe.

## Recommendations

### What to fix now (next cycle)

**R2: WorldlineSource → proper class hierarchy with correct name.**

`WorldlineSource` selects which worldline an observer projects. The
observer picks a worldline and a tick — it doesn't care about
frontiers (that's the writer's concern).

The three variants select three kinds of worldline:

| Current name | What it selects | Recommended name |
|---|---|---|
| `WorldlineSource` (base) | Worldline selector (abstract) | TBD — see design doc |
| `{ kind: 'live' }` | Canonical worldline (all writers merged) | TBD |
| `{ kind: 'coordinate' }` | Hypothetical worldline (specific writer tips) | TBD |
| `{ kind: 'strand' }` | Single writer's isolated worldline | TBD |

Naming is an open question in the cycle 0007 design doc. "Viewpoint"
was rejected as weird. "Frontier" was rejected because frontiers are
a writer concern. The right name should reflect "which worldline to
observe."

This is the cycle 0005 redo, but with correct conceptual grounding
and proper class design.

### What to fix soon

**R1: `Worldline` → `ReadHandle` or `ProjectionHandle`.**

The current `Worldline` class is a read facade over materialized state.
It is not a worldline. However, renaming it is a public API breaking
change that should be bundled with a major version bump. Defer to a
future cycle, but track as a backlog item.

**R3: `Observer` → grow into full observer.**

The Observer class correctly implements the projection component. The
gap is that it doesn't have basis, accumulation, or emission. This is
additive, not a rename — the class can grow without breaking changes.
Lower priority.

### What to leave alone

- All GREEN nouns — correct as-is.
- All YELLOW nouns — the gaps are real but not worth renaming. The
  concepts are correct; the implementations can grow.
- Y5 (`JoinReducer`) — internal name, not public API, not worth the
  churn.
