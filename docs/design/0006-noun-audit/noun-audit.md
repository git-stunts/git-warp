### Observer and Worldline Nouns

| Codebase Noun | Paper Concept | Alignment | Notes |
|---|---|---|---|
| `Worldline` (class) | Worldline — deterministic sequence `U₀ → U₁ → … → Uₙ` | **RED** | **The codebase `Worldline` is not a worldline.** In the papers, a worldline is the full sequence of states generated by ticks under a scheduling policy (WARP II, OG-I §2). In the code, `Worldline` is a thin read-handle that wraps a graph reference and a source descriptor. It calls `materialize()` and returns a snapshot. It has no history, no sequence, no BTR chain. It's a **projection handle**, not a worldline. |
| `WorldlineSource` (type) | Causal frontier `F` — the observer's causal boundary (OG-IV Def. 2) | **RED** | **Not a source of worldlines.** This type specifies WHERE to materialize — which frontier to project from. The three variants (live, coordinate, strand) are three ways of specifying a causal frontier. In OG-IV vocabulary, this is the `F` component of a replica `R = (S, F, Π)`. |
| `Observer` (class) | Observer — resource-bounded functor `O : Hist(U,R) → Tr` | **RED** | **Narrower than the paper concept.** In the papers, an observer is a full structural observer `S = (O, B, M, K, E)` with projection, basis, and accumulation (OG-I Def. 3). In the code, `Observer` is a filtered view of materialized state — glob match + expose/redact. It implements only the projection component `O`, not the full observer. The optics note confirms: "Observer is closer to the projection half of an optic." |
| `Aperture` (interface) | Aperture — the set of distinctions that survive projection (OG-I §4) | YELLOW | Correct concept, but the codebase `Aperture` is just `{ match, expose, redact }` — a glob filter. In OG-I, aperture is a formal measure over a distribution of distinctions (Def. 4 in OG-I). The codebase captures the intent but not the full formalism. Acceptable — the interface can grow. |
| `ObserverConfig` | Legacy alias for Aperture | GREEN | Just an alias. Correctly deprecated. |

## Recommendations

### What to fix now (next cycle)

**R2: WorldlineSource → proper class hierarchy with correct name.**

The `WorldlineSource` type specifies a causal frontier — where in the
DAG to project from. The three variants are:

| Current name | Theory concept | Recommended name |
|---|---|---|
| `WorldlineSource` (base) | Causal frontier specification | `Viewpoint` |
| `{ kind: 'live' }` | Current frontier (all writers, latest tips) | `LiveViewpoint` |
| `{ kind: 'coordinate' }` | Explicit frontier (pinned writer tips) | `CoordinateViewpoint` |
| `{ kind: 'strand' }` | Single-writer causal cone | `StrandViewpoint` |

**Why `Viewpoint`**: It's the observer's point of view — where they're
looking from. It specifies the frontier, not the worldline. It avoids
overloading `Frontier` (which is already a class in the codebase for
the octopus anchor frontier) and `Source` (which implies origin, not
vantage point).

This is the cycle 0005 redo, but with correct naming and proper class
design.

### What to fix soon

**R1: `Worldline` → `ReadHandle` or `ProjectionHandle`.**

The current `Worldline` class is a read facade over materialized state.
It is not a worldline. However, renaming it is a public API breaking
change that should be bundled with a major version bump. Defer to a
future cycle, but track as a backlog item.

**R3: `Observer` → grow into full observer.**

The Observer class correctly implements the projection component. The
gap is that it doesn't have basis, accumulation, or emission. This is
additive, not a rename — the class can grow without breaking changes.
Lower priority.

### What to leave alone

- All GREEN nouns — correct as-is.
- All YELLOW nouns — the gaps are real but not worth renaming. The
  concepts are correct; the implementations can grow.
- Y5 (`JoinReducer`) — internal name, not public API, not worth the
  churn.
